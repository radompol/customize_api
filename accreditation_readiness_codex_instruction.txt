CODEX INSTRUCTION
Project: External Accreditation Readiness API + Time Series Forecast + HTML Visualizer
Stack: Next.js + TensorFlow.js + Prisma + MySQL + CSV/XLSX Import

Build a Next.js external API system that ingests the exported accreditation readiness dataset from Excel/CSV, computes current readiness metrics, trains a TensorFlow.js time series forecasting model, and returns both:
1. API JSON responses for readiness and prediction
2. HTML visualizer pages for charts, tables, and readiness dashboards

The system must be scaffolded for future production deployment, but the first version may use file-based CSV upload/import and local database storage.

================================================================================
1. SYSTEM GOAL
================================================================================

The system will:
- accept an exported CSV/XLSX dataset from the accreditation view
- normalize and store the data
- compute readiness metrics per:
  - program
  - area
  - instrument
  - academic year
  - semester
- generate a current readiness level
- build a TensorFlow time series prediction model
- forecast future readiness trends
- expose all results through external REST API endpoints
- render an HTML visual dashboard in Next.js

================================================================================

A. Data ingestion
- upload CSV/XLSX

4. CORE BUSINESS FLOW
================================================================================

Step 1: Import dataset
The user uploads Excel/CSV exported from phpMyAdmin view.

Expected raw fields include at minimum:
- ap_id
- program
- acadYear
- semester
- area_id
- area_code
- area_description
- instrument_id
- requirement_description
- assigned_at
- assigned_modified_at
- assigned_status
- approvedOrder
- latest_uf_id
- latest_file_uploaded_at
- latest_file_version
- latest_file_status
- latest_filename
- latest_file_extension
- has_file
- total_files_uploaded
- total_status_logs
- revise_count
- reply_count
- total_comments
- non_empty_comments
- days_since_assignment
- days_since_latest_upload
- is_pending_flag

Optional future fields:
- deadline_date
- submission_date
- days_to_deadline
- days_overdue

Step 2: Normalize dataset
Store imported data as row-level requirement records.

Also generate snapshot records for:
- per program readiness
- per area readiness
- per semester readiness
- per academic year readiness

Step 3: Compute readiness score
Create a rule-based readiness score first.

Suggested component weights:
- Document completion = 30%
- Compliance status = 25%
- Quality / revisions / comments = 20%
- Aging / pending burden = 15%
- Timeline performance = 10%

If timeline fields are missing, redistribute weight proportionally.

Example scoring logic:
For each record:
- approved or compliant requirement gets higher points
- pending or revise lowers score
- more comments lowers score
- more revision cycles lowers score
- long aging lowers score
- no file lowers score

Then aggregate by:
- area
- program
- semester
- academic year

Step 4: Assign readiness level
Map score to labels:
- 0.85 to 1.00 -> Low Risk / Highly Ready
- 0.70 to 0.84 -> Moderate Risk / Nearly Ready
- 0.50 to 0.69 -> Medium Risk / Needs Attention
- below 0.50 -> High Risk / Not Ready

Also return:
- riskLevel
- priorityLevel
- warningFlags

Step 5: Build time series data
For TensorFlow time series, aggregate score by time bucket.

Minimum supported time axis:
- by import date
- by month
- by semester
- by academic year

If historical snapshots are not yet available, implement a snapshot table and seed the first snapshot from imported data.

Create time series sequences per:
- program
- area
- whole institution

Step 6: Train TensorFlow model
Use TensorFlow.js in server-side Node runtime.

Start with a simple sequence forecasting model:
- input = previous readiness scores across time
- output = next readiness score

Model may use:
- dense sequence model
- or simple LSTM if stable

Do not over-engineer first version.

Support:
- training on demand
- caching trained model
- retraining when new snapshots are imported

Step 7: Forecast readiness
Return:
- predicted next readiness score
- predicted next 3 periods
- trend direction
- confidence note
- risk forecast

Examples:
- improving
- stable
- declining
- volatile

6. API CONTRACT
================================================================================

POST /api/import
Purpose:
- upload CSV/XLSX
- parse and save records
- generate snapshots

Response example:
{
  "success": true,
  "batchId": 1,
  "rowsImported": 540,
  "snapshotsCreated": 28
}

GET /api/readiness/summary
Return institution-wide summary.

Response example:
{
  "success": true,
  "data": {
    "overallReadinessScore": 0.81,
    "readinessLabel": "Moderate Risk / Nearly Ready",
    "riskLevel": "Medium",
    "totalPrograms": 4,
    "totalRequirements": 1220,
    "completedRequirements": 950,
    "pendingRequirements": 270,
    "averageRevisionCount": 1.4,
    "averageCommentCount": 2.1
  }
}

GET /api/readiness/program/[program]
Return one program’s readiness summary.

GET /api/readiness/areas?program=BS%20Information%20Technology
Return readiness per area.

GET /api/readiness/trend?program=BS%20Information%20Technology
Return historical series for charting.

GET /api/readiness/forecast?program=BS%20Information%20Technology
Return forecast series.

Response example:
{
  "success": true,
  "data": {
    "program": "BS in Information Technology",
    "currentScore": 0.84,
    "forecast": [
      { "period": "2026-06", "score": 0.85, "riskLevel": "Medium" },
      { "period": "2026-07", "score": 0.87, "riskLevel": "Low" },
      { "period": "2026-08", "score": 0.88, "riskLevel": "Low" }
    ],
    "trendDirection": "Improving"
  }
}

GET /api/health
Return health status.

7. READINESS ENGINE RULES
================================================================================

Create lib/readinessEngine.ts.

Implement:
- computeRequirementScore(record)
- aggregateAreaReadiness(records)
- aggregateProgramReadiness(records)
- aggregateInstitutionReadiness(records)
- resolveRiskLevel(score)
- resolveReadinessLabel(score)

Example requirement-level logic:
Use normalized values:
- file presence
- latest status
- revision burden
- comment burden
- aging burden
- overdue burden if available

Principle:
- higher revisions/comments/aging reduce score
- approved/completed increases score
- pending/revise lowers score

Do not hardcode only one status vocabulary; make status mappings configurable in constants.ts.

================================================================================
8. TENSORFLOW TIME SERIES MODULE
================================================================================

Create lib/forecasting.ts.

Use TensorFlow.js server-side.

Tasks:
- extract historical readiness scores from readiness_snapshots
- sort by time ascending
- normalize values
- build lookback windows
- train model
- forecast next N periods

Minimum functions:
- buildTimeSeriesDataset(snapshots, lookback)
- trainForecastModel(series)
- predictNextPeriods(model, seedWindow, periods)
- classifyTrendDirection(actualAndForecastedValues)

Guidance:
Use small sequence input first:
- lookback = 3 or 6 periods

Return forecast even if data is limited, but mark:
- confidence_note = "Low historical depth" when snapshots are few

Important:
If historical data is too small:
- do not fail hard
- fallback to moving-average or trendline projection
- still return usable API data
